# PyPI-First Distribution: Keep Submodules, Retire Git URL as Stable Install Path

## Problem Frame

<!-- BEGIN:problem-frame -->
autoharness 1.4.1 users could not upgrade via `uv tool upgrade autoharness` when installed from `git+https://github.com/softwaresalt/autoharness.git`. Root cause: `uv` recursively initialises all Git submodules during a git-URL install/upgrade. The six `references/` submodules are large community repos; on Windows, deep path traversal inside the `uv` cache causes the upgrade to fail silently or error out. A prior working-tree attempt resolved this by removing `.gitmodules` and all six submodule gitlinks and replacing them with a `references/README.md` index file. The operator has explicitly rejected that approach: the submodules are intentional in-repo references used during template curation and must be preserved as-is. The chosen direction is to solve the user-facing upgrade problem at the distribution layer — publish autoharness to PyPI so that `uv tool install autoharness` and `uv tool upgrade autoharness` target the PyPI wheel (which does NOT include submodule content) rather than the Git repository.
<!-- END:problem-frame -->

## Options

<!-- BEGIN:options -->
**Option A — Remove submodules (rejected prior attempt)**
Remove `.gitmodules` and all six `references/` gitlink entries. Replace with a `references/README.md` link index. Pros: eliminates the uv submodule problem for both Git and PyPI installs. Cons: destroys the operator's in-repo reference browsing workflow; contradicts explicit operator instruction; `references/README.md` is an inferior substitute for live submodule content during template curation.

**Option B — Keep submodules + publish to PyPI (operator's chosen direction)**
Leave `.gitmodules` and all six `references/` submodule registrations exactly as committed at HEAD. Publish autoharness as a wheel to PyPI on every tagged release. End-users install/upgrade from PyPI (`uv tool install autoharness`, `uv tool upgrade autoharness`) — no Git clone, no submodule traversal. The Git URL (`git+https://...`) is documented as an explicit snapshot/developer path, not the stable distribution channel. The PyPI wheel packages `templates/`, `schemas/`, `.github/agents`, `.github/skills`, etc., but explicitly excludes `references/` content (already the case in pyproject.toml). The release CI checkout step uses `submodules: false` so GitHub Actions never traverses submodule trees during the publish job.

**Option C — Shallow submodule clone in .gitmodules**
Set `shallow = true` or `fetchRecurseSubmodules = false` in `.gitmodules` to limit clone depth. Pros: keeps submodules, reduces clone cost. Cons: does not eliminate the uv submodule problem on Windows; uv still attempts recursive init regardless of depth hints; partial fix at best.
<!-- END:options -->

## Chosen Direction

<!-- BEGIN:chosen-direction -->
**Option B** — keep submodules, publish to PyPI. This is the operator's explicit instruction and is the architecturally correct separation: the reference library is a developer tool (submodules, only initialised manually by maintainers), while the distribution artefact is the PyPI wheel (zero submodule content, upgradeable with a single `uv tool upgrade autoharness` on any platform). The working-tree already contains most of the PyPI pipeline work; the primary action is to undo the submodule removals, harden the one gap in the release workflow (`submodules: false` on checkout), revise the documentation and one contrary test to match the submodule-kept model, and commit the aligned subset of working-tree changes.
<!-- END:chosen-direction -->

## Open Questions

<!-- BEGIN:open-questions -->
1. **Version bump vs. amend 1.4.1**: The working-tree is still at 1.4.1. The PyPI distribution changes are substantive enough to warrant a separate release entry. Does the operator want to bump to 1.4.2 (or 1.5.0) for the PyPI-first release, or amend 1.4.1 CHANGELOG and publish 1.4.1 to PyPI for the first time? (Recommendation: amend 1.4.1 and treat it as the first PyPI-published release — the version already reflects the intent.)
2. **PyPI Trusted Publisher**: Must be configured before the release workflow can publish. This is a human action on pypi.org that cannot be automated by Ship. Blocking if not done before the first tag push.
3. **Submodule initialisation for maintainers**: The submodules are registered but not initialised in most clones. The revised reference-library.md should document that maintainers run `git submodule update --init references/&lt;name&gt;` manually when they need a working copy of a specific reference repo.
<!-- END:open-questions -->
