# Safe dark factory mode deliberation

## Problem Frame

<!-- BEGIN:problem-frame -->
Stash E6DA0DCC asks Orchestrator to recognize prompts such as "Run pipeline in dark mode" as a bounded instruction to run the Stage -> Ship harness autonomously, including pre-authorized PR merge approval. Stash 67023DBA asks for safe full dark-factory mode that relies primarily on local review/readiness, reduces dependence on GitHub Copilot shadow reviews, and adds an ATV-inspired brainstorm step before autonomous handoff. The decision must preserve P-001, P-014, P-016, branch protection, safety approval boundaries, and operator-visible telemetry.
<!-- END:problem-frame -->

## Options

<!-- BEGIN:options -->
Option A — Minimal trigger alias: add a trigger phrase that runs the existing pipeline with no new policy. Low effort, but unsafe because merge approval, admin fallback, review authority, and dark-mode audit semantics remain implicit.

Option B — Policy-first bounded dark factory mode: define a new autonomy contract/policy first, then weave trigger handling, brainstorm handoff, local review, merge approval, admin fallback, telemetry, and verification surfaces through Orchestrator/Ship/skills. More work, but produces safe execution-ready semantics.

Option C — Local-review migration only: reduce shadow-review reliance while leaving dark factory mode out of scope. Useful but misses the operator's autonomous pipeline goal.

Option D — Fully unchecked autonomous mode: let the agent run and merge without additional gates. Rejected as incompatible with P-014, P-016, branch protection, and operator trust.
<!-- END:options -->

## Chosen Direction

<!-- BEGIN:chosen-direction -->
Adopt Option B. The first shipment must define the dark-factory autonomy contract before any Orchestrator or Ship implementation depends on it. Dark mode is an explicit, exact trigger with bounded scope; it pre-authorizes routine PR merge approval only after local review readiness, CI, branch/worktree, and merge-strategy gates pass. Copilot/GitHub hosted review remains optional advisory shadow review by default. Admin fallback is explicitly modeled and logged; it is not silently implied unless the policy says the trigger grants it for a bounded shipment.
<!-- END:chosen-direction -->

## Open Questions

<!-- BEGIN:open-questions -->
Ship implementation should decide the exact policy ID after P-016, likely P-017. Ship should also verify whether the repo's branch protection allows admin bypass from available credentials and encode the fallback as a fail-closed path if not explicitly allowed. The brainstorm task should decide whether to create a new brainstorm skill template or expose an alias/wrapper around deliberate while preserving autoharness' existing deliberation contract.
<!-- END:open-questions -->
