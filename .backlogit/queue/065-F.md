<!-- BEGIN:description -->
The pre-execution T-shirt sizing gate (`src/autoharness/gates/sizing.py`) estimates a size for unsized tasks and writes it back with `backlogit update <task_id> --size <XS|S|M|L|XL>`. That write-back currently fails in this workspace: the backlogit `task` and `subtask` artifact types in `.backlogit/header-def.yaml` define only `priority` and `status` — there is no `size` field — so backlogit rejects the update with `artifact type "task" does not define a size field`. The gate fails open (never blocks a task), so the estimated size is silently discarded and never persists. `backlogit update --size` is a first-class native flag (values XS, S, M, L, XL), so the remedy is to define a `size` field on the executable work-item types.
<!-- END:description -->

<!-- BEGIN:dod -->
- `.backlogit/header-def.yaml` `task` and `subtask` types declare a `size` enum field (values XS, S, M, L, XL), mirroring the existing `priority` enum shape.
- `backlogit sync` runs clean and `backlogit update <task-id> --size M` persists without the "does not define a size field" error.
- `autoharness gate size <unsized-task>` reports `wrote size: <X>` and the size is visible on the artifact.
- `pytest tests/test_gate_size_cli.py tests/test_gates_sizing.py` remains green.
<!-- END:dod -->

<!-- BEGIN:goals -->
- Define a `size` field on the backlogit `task` (and `subtask`) types so `backlogit update --size` succeeds instead of being rejected.
- Make the sizing gate's write-back actually persist a T-shirt size on unsized tasks (end-to-end, not just in mocked tests).
- Keep the change scoped to the dogfood backlogit config; no autoharness product template ships a `header-def`, so no template change is required and the gate's fail-open behavior is preserved for workspaces that lack the field.
<!-- END:goals -->
