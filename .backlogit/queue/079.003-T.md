title: Persisting v1.1 telemetry fields in local sinks
updated_at: 2026-07-14T17:59:23.418-07:00

Persist the new `ExecutionEpoch` v1.1 epoch-level roll-up fields and optional `WorkSizingSnapshot` payload through the existing local sinks without changing fail-open behavior. SQLite must support time-series queries over persisted epoch fields while JSONL keeps exact serialized epoch records. This task does not add a `ToolTelemetryEvent` sink, queryable event store, or live backlogit sizing adapter.

<!-- BEGIN:acceptance-criteria -->
- SQLite schema migration is idempotent and adds queryable columns or JSON columns for session/task/feature/shipment/phase/git correlation, token economics, provenance maps, offload counts, route kinds, expected/observed/missing per-tool gap maps, degraded counts, index freshness counts, and the optional `WorkSizingSnapshot`.
- A migration test starts from the current pre-079-F SQLite schema and proves legacy rows remain readable while new v1.1 rows persist without dropping older fields.
- Legacy v1.0 rows normalize to v1.1 reporting defaults with all metric provenance marked `unavailable` for every metric not established as observed, including pre-existing v1.0 economic fields; no hybrid `schema_version: 1.0.0` plus v1.1 fields is emitted.
- JSONL output contains the full v1.1 epoch record through `ExecutionEpoch.to_record`, including the already captured pre-execution `WorkSizingSnapshot` when provided, without raw tool-output bodies or granular event payloads.
- Persistence is first-write immutable per `epoch_id`: SQLite must use insert-only/conflict-detect behavior rather than `INSERT OR REPLACE`, store or derive a canonical payload digest, treat identical replay as idempotent success without replacement, and reject/diagnose conflicting replay while preserving the first accepted payload.
- JSONL replay handling avoids duplicate appends for identical payloads and surfaces conflicting later payloads for the same `epoch_id`; reader-level conflict handling is completed in 079.005-T.
- Partial fail-open sink-write tests simulate one sink accepting an epoch while another fails: retry with the same `epoch_id` and matching digest repairs only the missing sink, while retry with a different digest writes nothing and reports a conflict.
- `record_epoch` remains fail-open for sink availability failures: failures are summarized and never block task completion, but identity/schema errors and conflicting immutable replays are explicit diagnostics rather than silent replacement.
<!-- END:acceptance-criteria -->
