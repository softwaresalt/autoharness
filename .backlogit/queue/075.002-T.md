Add _check_capability_pack_enforcement to src/autoharness/verify_workspace.py. Expectedness driven by the set of ENABLED retrieval-enforced packs (agent-engram/graphtor-docs). NOTE: the existing installed_packs at verify_workspace.py ~2948-2951 is built from manifest.capability_packs OR config.capability_packs, so it is NOT independent of manifest membership — a nonempty manifest that omits a pack while config still enables it would silently disable this check. Therefore compute the enabled retrieval-enforced set from the UNION of manifest.capability_packs and config.capability_packs (config is authoritative for "enabled") so dropping a pack from the manifest's list cannot suppress enforcement. When >=1 retrieval-enforced pack enabled, INDEPENDENTLY require: (a) instruction file exists; (b) manifest-listed with a checksum that MATCHES the installed file content; the check itself FAILS (not merely warns) on a missing/empty checksum OR a mismatch (the generic scan only records `checksum-untracked` / `user-modified` as warnings — verify_workspace.py ~2988-3013 — so this check enforces checksum integrity independently); (c) rendered route rows represent EXACTLY the enabled set (via stable markers, not loose substring); (d) key safeguard invariant markers present (pack deferral, direct-search exemptions, per-phase health reuse, internal-no-public-web). Also assert frontmatter applyTo '**' and no unresolved placeholders. When NO retrieval pack enabled: no-op ONLY if file+manifest both absent; else FAIL as orphaned overlay. The retrieval-enforced pack set is a module constant drift-guarded against the registry in 075.003-T (verifier stays self-contained; it does not read autoharness_home registry at target verify-time). Wire in after copilot check. Lives in verify_workspace, not gates/ (import boundary).
