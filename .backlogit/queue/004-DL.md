# Engram First-Line Integration — autoharness Dogfood Workspace

## Problem Frame

<!-- BEGIN:problem-frame -->
The autoharness dogfood workspace has Engram registered as an MCP server in `.mcp.json` (root level) and partially references it in `.github/copilot-instructions.md` via an `<!-- engram:start -->` block. However, the harness treats Engram as absent: `workspace-profile.yaml` shows `agent_engram.detected: false / mcp_configured: false / recommended: false`, neither `config.yaml` nor `harness-manifest.yaml` lists `agent-engram` in `capability_packs`, no `agent-engram.instructions.md` is installed in `.github/instructions/`, and neither `stage.agent.md` nor `ship.agent.md` contains any Engram-first search or impact-analysis weaving.

Root causes:
1. **Detection gap**: workspace-discovery SKILL.md Step 1.5d checks `.vscode/mcp.json` only. This workspace uses `.mcp.json` at the repo root (the VS Code MCP convention), which the detection does not scan.
2. **instruction_markers miss**: `.github/copilot-instructions.md` contains an `<!-- engram:start -->` block listing `set_workspace`, `query_memory`, `unified_search`, etc., but the detection signal patterns did not match any of these to produce instruction_markers entries.
3. **Pack not installed**: The `templates/instructions/agent-engram.instructions.md.tmpl` template exists but has never been rendered/installed into `.github/instructions/`. The overlay was never applied to the dogfood harness.
4. **Metadata inconsistency**: `config.yaml` and `harness-manifest.yaml` do not list `agent-engram`, so tune-harness drift detection will never flag the absence as drift.
5. **Agent weaving absent**: stage.agent.md and ship.agent.md have zero Engram-first search weaving. Neither agent calls `get_workspace_status`, `unified_search`, or `impact_analysis` before planning or build work.
6. **Daemon startup failure**: `engram workspace-status` currently fails with "Daemon failed to reach Ready state within 30000ms". Without an installed instruction teaching retry + CLI fallback, agents will either loop on startup or bypass Engram entirely.
<!-- END:problem-frame -->

## Options

<!-- BEGIN:options -->
**Option A — Minimal patch: update profile metadata only**
Update `workspace-profile.yaml`, `config.yaml`, and `harness-manifest.yaml` to reflect detected=true / recommended=true. Leave agents and instructions unchanged.
Pros: Very low blast radius. Cons: Does not install the instructions file; does not weave Engram into agents; does not fix the detection root cause for future re-runs; copilot-instructions.md Engram block remains disconnected from installed harness artifacts.

**Option B — Instructions-only install**
Install `agent-engram.instructions.md` from the existing template and update metadata. Do not weave agents.
Pros: Instruction guidance available. Cons: Stage/ship agents still won't call Engram tools; weaving is half-done; detection root cause unfixed; future tune-harness re-runs will re-detect as absent if discovery is not fixed.

**Option C — Comprehensive overlay: fix detection + install overlay + weave agents + document daemon fallback**
1. Fix workspace-discovery SKILL.md to also check `.mcp.json` (root) and recognise `<!-- engram:* -->` markers in copilot-instructions.md as instruction_markers.
2. Update dogfood metadata files (workspace-profile, config, manifest) to truth.
3. Install `agent-engram.instructions.md` from template.
4. Weave Engram-first search and impact-analysis protocol into stage.agent.md and ship.agent.md.
5. Expand the copilot-instructions.md Engram block to reference the installed instructions file and document daemon startup retry + CLI fallback expectation.
6. Verify install-harness SKILL.md overlay contract for agent-engram is complete (it currently exists as an example stub — harden into a full contract).
Pros: Fixes the root cause; installs the full pack correctly; makes Engram first-line for both agents; unblocks future tune-harness; daemon failure becomes a handled degraded mode rather than a binary blocker. Cons: Wider diff, six distinct concerns, requires a planning branch.
<!-- END:options -->

## Chosen Direction

<!-- BEGIN:chosen-direction -->
**Option C — Comprehensive overlay.** This is the correct approach for a harness framework that dogfoods itself. Partial fixes leave the harness in an inconsistent state that will silently drift further. The detection root-cause fix (`.mcp.json` scan path) is essential — without it, the next tune-harness run will re-detect Engram as absent and propose rolling back any manual edits.

The work is decomposed into six tight workstreams (WS-A through WS-F), each targeting a single artifact class, within the 2-hour rule. No workstream requires writing application code; all are template authoring, instruction authoring, or metadata updates — all within Stage's planning domain for a planning branch, and within Ship's implementation domain for the actual mutations.

Dependencies: WS-B (metadata) must precede WS-C (install instructions); WS-C must precede WS-D (agent weaving references instructions); WS-A (discovery fix) is parallel to WS-B/C/D. WS-E (copilot-instructions) and WS-F (install-harness overlay contract) are parallel to each other and to WS-D.
<!-- END:chosen-direction -->

## Open Questions

<!-- BEGIN:open-questions -->
1. **Daemon startup failure**: Should the harness document a specific retry count before declaring the daemon unavailable (template already says "retry once"), and should it specify the engram CLI command for degraded-mode indexing? The installed instructions template already covers this; confirming operator does not want a custom override.
2. **`.mcp.json` vs `.vscode/mcp.json` normalization**: The workspace-discovery fix should scan both locations. Should it also scan `.cursor/mcp.json` for Cursor compatibility? Recommend yes — aligns with environment-agnostic principle (AGENTS.md Rule 3).
3. **Stage/ship agent weaving depth**: The overlay contract example lists "indexed search routing" targeting "research / planning / build workflows". For this dogfood workspace, the agents are the workflow layer. Confirm the weaving covers: (a) session-start `get_workspace_status` / `set_workspace`, (b) pre-planning `unified_search` / `impact_analysis`, (c) fallback documentation when daemon is unavailable.
4. **Version alignment**: The installed instructions template references `(032-S)`, `(033-S)`, `(034-S)` as version tags for internal Engram changes. These are metadata notes in the template, not `{{VARIABLE}}` placeholders, so they pass through as-is. Confirm operator is comfortable with these tags remaining in the installed file.
5. **Engram `create_task` / `update_task` duplication**: copilot-instructions.md currently lists `create_task` and `update_task` in the Engram tool table. These conflict with backlogit as the canonical task tracker (P-001 scope isolation). The installed `agent-engram.instructions.md` template does not mention those tools. The copilot-instructions.md Engram block should be updated to drop `create_task` / `update_task` from the tool table and use backlogit for all task tracking.
<!-- END:open-questions -->
