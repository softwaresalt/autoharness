Deliver a complete PyPI-backed release and distribution pipeline for autoharness while preserving the `references/` Git submodules as in-repo developer references. Solves the `uv tool upgrade` failure on Windows (submodule-recursive git installs) by making PyPI the canonical stable distribution channel. The Git URL is retained as an explicit snapshot/developer path only.

Deliberation: 003-DL
Root cause: uv recursively initialises submodules during git-URL installs; six large community repos in references/ cause Windows uv cache path failures.
Chosen direction: keep submodules, publish to PyPI — architectural separation between developer reference library (submodules, manually initialised) and end-user distribution artefact (PyPI wheel, no submodule content).

Acceptance criteria:
- All six references/ submodules registered in .gitmodules and committed at HEAD (no deletions)
- release.yml publishes wheel+sdist to PyPI on every v* tag push with OIDC (id-token: write), smoke-tests the published package, then creates/updates GitHub Release
- release.yml checkout step has submodules: false (CI never traverses submodule trees)
- docs/getting-started.md, README.md, cli.py help text all recommend `uv tool install autoharness` (PyPI) as canonical path; git URL documented as snapshot/dev only
- Migration note present in docs for existing git-URL-installed users
- tests/test_verify_workspace.py: two new passing tests (release workflow structure, install guidance) committed; one contrary test (no-.gitmodules assertion) replaced with submodule-registration assertion
- docs/reference-library.md consistent with submodule-kept + PyPI-first model
- CHANGELOG 1.4.1 amended with PyPI distribution pipeline entry (or version bumped — operator confirms)
