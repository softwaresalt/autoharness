<!-- BEGIN:description -->
Define a `size` field on the backlogit `task` and `subtask` artifact types in `.backlogit/header-def.yaml` so the sizing gate's write-back (`backlogit update <task_id> --size <X>`) succeeds and persists. Follow the existing `priority` enum declaration shape. Run `backlogit sync`, then verify write-back both via the raw CLI and via the `autoharness gate size` end-to-end path.
<!-- END:description -->

<!-- BEGIN:implementation-notes -->
- Declaration pattern: `type: enum` with a `values:` list of XS/S/M/L/XL and NO `default`. Do NOT mirror `priority`'s `default: medium`: backlogit applies field defaults at creation time (as the existing `priority: medium` shows), so a `size` default would pre-size every new task. The gate only estimates when no size exists (`sizing.py` `_existing_size`/`size_task`), so a defaulted size would make every task short-circuit to `skipped-existing` and permanently bypass estimation. Keep `size` optional and unset by default.
- `backlogit update --size` is "body-preserving, mutually exclusive with other field flags" â€” confirm size updates do not clobber body sections.
- Scope: dogfood config only. autoharness ships NO `header-def` template (verified: `templates/backlog` holds only registries; `templates/instructions` holds backlogit instruction templates), so the product needs no template change and the gate's fail-open path stays as the safety net for workspaces lacking the field.
- The gate only sizes tasks, so `task` + `subtask` coverage is sufficient; if `feature` is intentionally excluded, note it.
