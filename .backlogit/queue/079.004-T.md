title: Detecting expected-versus-actual tool usage gaps
updated_at: 2026-07-13T22:29:26.4862208-07:00
---

Implement the expected-vs-actual tool usage gap signal over epoch-level roll-up aggregates. A gap is counted when policy expected an indexed/routed tool but epoch aggregates show raw search/read, degraded fallback, unavailable/stale route, or no observed invocation at all. This task must not assume a queryable event stream.

<!-- BEGIN:acceptance-criteria -->
- Unit tests cover expected Engram route used, expected route missed and raw read used, stale/unavailable route fallback, expected-but-no-tool-invoked, and no-gap cases when no route was expected.
- Gap inputs are explicit epoch aggregates: `expected_tool_counts`, `observed_tool_counts`, `missing_expected_tool_counts`, route-kind counts, freshness/degraded counts, and safe evidence references when available.
- Gap counts roll up to `ExecutionEpoch` fields and can be queried from persisted epoch sink data without any event table.
- Per-tool gap rate is defined deterministically as `missing_expected_tool_counts[tool] / expected_tool_counts[tool]` when expected count is nonzero.
- The implementation does not enforce routing policy itself; it only records and reports evidence.
<!-- END:acceptance-criteria -->
