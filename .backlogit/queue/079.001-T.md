Implement the ratified `ExecutionEpoch` v1.1 additive token-economics and epoch-correlation fields from docs/decisions/2026-07-13-telemetry-metrics-reporting-ownership.md. Keep this scoped to the epoch model and model tests: token consumption vs generation, cumulative context-area totals, per-metric provenance maps, and persisted correlation fields needed for task/session/phase reporting.

<!-- BEGIN:acceptance-criteria -->
- Tests are written first for v1.1 serialization, generation-vs-consumption separation, per-metric unavailable provenance defaults, and no cumulative-token double-counting.
- `ExecutionEpoch.SCHEMA_VERSION` is bumped to `1.1.0`; legacy v1.0 payload reads normalize to v1.1 with additive fields marked `unavailable`, never a hybrid `schema_version: 1.0.0` record containing v1.1 fields.
- Add root correlation fields exactly as assigned by the decision matrix: `workspace_id`, `session_id`, `agent_role`, `phase`, `backlog_item_id`, `shipment_id`, `branch`, and `commit_sha` so every promised reporting slice is backed by a persisted field.
- `EconomicPayload` adds token/cost fields exactly as assigned by the decision matrix: `cached_input_tokens`, `cumulative_input_tokens`, `cumulative_output_tokens`, `context_tokens_before`, `context_tokens_after`, `context_area_tokens`, `avoided_read_estimated_tokens`, `tool_output_estimated_tokens`, `metric_sources`, and `metric_quality`; v1.0 records default additive metric provenance to `unavailable`.
- Cumulative fields are final epoch-close totals or max/final observed cumulative values, not values summed across events; tests assert two running cumulative samples do not double-count.
<!-- END:acceptance-criteria -->
