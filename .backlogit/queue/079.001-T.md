Implement the ratified `ExecutionEpoch` v1.1 additive token-economics, epoch-correlation, and `WorkSizingSnapshot` fields from docs/decisions/2026-07-13-telemetry-metrics-reporting-ownership.md. Keep this scoped to the epoch model and model tests: token consumption vs generation, cumulative context-area totals, per-metric provenance maps, persisted correlation fields needed for task/session/feature/shipment reporting, and the optional immutable sizing snapshot shape.

<!-- BEGIN:acceptance-criteria -->
- Tests are written first for v1.1 serialization, generation-vs-consumption separation, per-metric unavailable provenance defaults, and final cumulative-token preservation.
- The module-level `SCHEMA_VERSION` constant in `src/autoharness/telemetry/epoch.py` (the dataclass default for `schema_version`) is bumped to `1.1.0`; `ExecutionEpoch` has no class-level `SCHEMA_VERSION` attribute. Legacy v1.0 payload reads normalize to v1.1 with every metric that cannot be established as observed marked `unavailable` or `not_applicable`, never a hybrid `schema_version: 1.0.0` record containing v1.1 fields.
- Add root correlation fields exactly as assigned by the decision matrix: `workspace_id`, `session_id`, `agent_role`, `phase`, `backlog_item_id`, `feature_id`, `shipment_id`, `branch`, and `commit_sha` so every promised reporting slice is backed by a persisted field; for task epochs `backlog_item_id` equals the existing required `task_id`, and v1.0 normalization copies `task_id` into `backlog_item_id` when absent. Replayable v1.1 task-close construction rejects missing, unparsable, empty, or whitespace-only `epoch_id` values unless a pre-execution context from 079.014-T supplies the canonical UUID ID.
- `EconomicPayload` adds token/cost fields exactly as assigned by the decision matrix: `cached_input_tokens`, `cumulative_input_tokens`, `cumulative_output_tokens`, `context_tokens_before`, `context_tokens_after`, `context_area_tokens`, `avoided_read_estimated_tokens`, `tool_output_estimated_tokens`, `metric_sources`, and `metric_quality`; v1.0 records default all unobserved metric provenance to `unavailable` or `not_applicable`, including pre-existing v1.0 economic fields whose numeric `0` values came from defaults rather than observation.
- Every populated epoch metric has a same-named entry in both `metric_sources` and `metric_quality`; tests reject populated metrics that lack provenance entries.
- Add an optional `WorkSizingSnapshot` payload exactly as assigned by the decision matrix: nullable task/feature/shipment planned size labels, `snapshot_at`, `snapshot_boundary`, per-level `sizing_sources`, per-level planned-estimate `sizing_source_revisions`, per-level `sizing_ruleset_versions`, feature child task count/histogram/hash, and shipment manifest task count/histogram/hash.
- `snapshot_boundary` is the fixed contract value `pre_execution`; `snapshot_at` is the time that host captured the snapshot after planning/manifest assembly and before implementation/tool work began, not epoch close.
- `WorkSizingSnapshot` tests assert labels are ordinal buckets only, no numeric point fields or implicit label-to-point mapping exists, and unavailable hierarchy/sizing data remains explicit `null`/`unavailable` rather than fabricated values.
- `WorkSizingSnapshot` model tests assert feature/shipment counts, histograms, and membership hashes are internally consistent when membership is known: count equals the cardinality of the canonical unique ID set and equals the sum of the `histogram` values plus the `unsized` count, with `skipped` unresolved IDs excluded from both; duplicate IDs are not counted twice. There is no `unavailable` histogram bucket.
- Cumulative fields are final epoch-close totals or max/final observed cumulative values; tests assert the host-supplied final cumulative total is preserved on round-trip and is never re-derived or summed. Cross-event double-count assertions belong to 084-F event→epoch composition.
<!-- END:acceptance-criteria -->
