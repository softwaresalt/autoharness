# No parallel branches/worktrees policy

## Problem Frame

<!-- BEGIN:problem-frame -->
Stash CE080560 asks autoharness to adopt a fundamental operating rule: agents must not work on parallel branches or maintain one local branch plus one or more parallel worktrees, because remote operators cannot reliably manage branch-change state across concurrent branches/worktrees. The only exception should be explicit Stage-owned spike/research worktrees used for staging investigation, not implementation or Ship execution.
<!-- END:problem-frame -->

## Options

<!-- BEGIN:options -->
Option A — Fold the rule only into P-011 Branch-Before-Mutation. Small, but it reaches Ship branch creation only and misses Orchestrator pipelining, Stage spike exceptions, and workspace-level constitutional guidance.

Option B — Add a first-class workflow policy plus constitutional/workflow guidance. Covers the invariant globally, lets P-011 reference the stronger no-parallel-worktrees gate, and gives Ship a clear implementation target. Requires coordinated template updates.

Option C — Create a detached instruction only. Easy to install, but risks drift from existing branch/workflow policies and may not be consulted at the branch creation gate.

Option D — Disable all Stage/Ship pipelining. Too broad: Stage can still safely prepare planning artifacts while Ship waits, and Stage spike/research worktrees are an explicit exception.
<!-- END:options -->

## Chosen Direction

<!-- BEGIN:chosen-direction -->
Adopt Option B. Add a new first-class policy (next ID after P-015, likely P-016) and weave it through the constitution, workflow policy registry, concurrency guidance, Ship branch gate, Stage spike/worktree guidance, and Orchestrator pipelining constraints. Treat P-011 as the branch-creation implementation hook, but keep the new rule broader than P-011 because it governs parallel worktrees/branches across agents and operating modes.
<!-- END:chosen-direction -->

## Open Questions

<!-- BEGIN:open-questions -->
Ship should decide exact detection mechanics for the P-016 worktree gate. Suggested check: inspect `git worktree list --porcelain` before branch creation/claim and halt when more than the current worktree is attached to an implementation branch, except an explicitly named/recorded Stage spike/research worktree that is read-only, time-boxed, and not used by Ship.
<!-- END:open-questions -->
