Add declarative content to orchestrator.agent.md.tmpl that defines the elective agent concept and makes them discoverable to operators.

**Changes** (all in `templates/agents/orchestrator.agent.md.tmpl`):

1. **New `## Elective Agents` section** (insert after `## Role`, before `## Environment Agnostic`):
   - Explain the elective agent concept: invoked on-demand by operator request, NOT part of the default Stage→Ship pipeline
   - Document Auto-MergeInstall: discovers a target workspace and installs a customized harness from templates. Reference: `.github/agents/auto-mergeinstall.agent.md`
   - Document Auto-Tune: detects drift in an installed harness and proposes updates. Reference: `.github/agents/auto-tune.agent.md`
   - Clarify that elective agents follow the same subagent model as Stage/Ship (Orchestrator invokes them as depth-1 subagents)
   - Note: these agents modify harness artifacts in the target workspace, so they must not run concurrently with active Ship work

2. **Trigger Phrases table update** (existing `## Trigger Phrases` section):
   - Add row: `install harness` / `install {workspace}` → Route to Auto-MergeInstall for workspace discovery and harness installation
   - Add row: `tune harness` / `tune {workspace}` → Route to Auto-Tune for drift detection and harness update proposals
   - Add row: `assess harness` → Report installed harness state without acting (analogous to `assess state` for pipeline)

3. **Model Routing table update** (existing `## Model Routing` section):
   - Add Auto-MergeInstall entry: Tier 3 (Frontier) — installation requires deep template composition reasoning
   - Add Auto-Tune entry: Tier 2 (Standard) — tuning is more mechanical, checksumming and diff-driven
