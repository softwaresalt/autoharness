title: Detecting expected-versus-actual tool usage gaps
updated_at: 2026-07-24T03:49:26.3429785Z
---

Implement the expected-vs-actual tool usage gap signal over epoch-level roll-up aggregates. A `tool_gap_count` gap is counted strictly when policy expected an indexed/routed tool and that expected tool was not invoked (missing invocation). Raw search/read may explain the miss; degraded fallback or unavailable/stale route where the expected tool was invoked is reported via separate diagnostic counters, not summed into `tool_gap_count`. This task must not assume a queryable event stream.

<!-- BEGIN:acceptance-criteria -->
- Unit tests cover expected Engram route used, expected route missed and raw read used, stale/unavailable route fallback where the expected tool was invoked as diagnostics-only, expected-but-no-tool-invoked gap, and no-gap cases when no route was expected.
- Gap inputs are explicit epoch aggregates: `expected_tool_counts`, `observed_tool_counts`, `missing_expected_tool_counts`, `route_kind_counts`, freshness/degraded diagnostic counts, and the scalar gap totals defined in the decision matrix; degraded/stale diagnostics are surfaced alongside gaps but are not summed into `tool_gap_count`.
- Gap counts roll up to `ExecutionEpoch` fields and can be queried from persisted epoch sink data without any event table.
- Per-tool gap rate is defined deterministically as `missing_expected_tool_counts[tool] / expected_tool_counts[tool]` when expected count is nonzero.
- The implementation does not enforce routing policy itself; it only records and reports evidence.
<!-- END:acceptance-criteria -->
