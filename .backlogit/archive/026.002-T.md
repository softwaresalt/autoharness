Add procedural routing logic to orchestrator.agent.md.tmpl that teaches the Orchestrator how and when to dispatch elective agents.

**Changes** (all in `templates/agents/orchestrator.agent.md.tmpl`):

1. **New routing step in `## Required Steps`** — insert as `### Step 1E: Route to Elective Agent (when operator requests install or tune)`:
   - **Trigger**: Operator explicitly requests harness installation or tuning (via trigger phrases or intent inference)
   - **Precondition gate**: No active Ship shipment. Elective agents modify harness artifacts in the target workspace, which could conflict with Ship's branch work. If a Ship shipment is active, inform the operator and defer until Ship completes.
   - **Dispatch logic**:
     - For install requests: invoke Auto-MergeInstall subagent, passing the target workspace path and any operator preferences
     - For tune requests: invoke Auto-Tune subagent, passing the target workspace path
   - **Output handling**: Record the installation/tuning result. If the elective agent produces stash entries (e.g., Auto-Tune identifies issues that need backlog work), feed them back to Step 0 for pipeline processing.
   - **Failure handling**: Surface failure to operator; do not retry elective agents automatically (they operate on external workspaces where state may be uncertain)
   - **Intercom integration**: When agent-intercom is installed, broadcast elective agent invocation and completion

2. **Update `## Concurrency Control` section**:
   - Add paragraph: "Elective agents (Auto-MergeInstall, Auto-Tune) must not run concurrently with an active Ship shipment. These agents modify harness artifacts in target workspaces, and Ship may also be modifying generated artifacts on its feature branch. The Orchestrator must check for active Ship work before dispatching an elective agent and defer if Ship is in-flight."
   - Clarify: Elective agents CAN run while Stage is planning (Stage only touches backlog artifacts, not harness artifacts in target workspaces)

3. **Update `## Subagent Depth` section**:
   - Extended the depth model with a structured table: Orchestrator (0) → Pipeline or elective agent (1) → skills (2) → review personas (3)
   - Documents that Auto-MergeInstall reaches max depth 2 (workspace-discovery and install-harness skills)
   - Documents that Auto-Tune may reach depth 3 (Orchestrator → Auto-Tune → verify-harness → review persona)
   - References the `subagent_depth: 2` declared in both agent definitions

4. **Update `## Remote Operator Integration (agent-intercom)` table**:
   - Add row: Elective agent routed → broadcast info → `[ORCHESTRATOR] {agent_name} subagent invoked — {action}`
   - Add row: Elective agent complete → broadcast success → `[ORCHESTRATOR] {agent_name} complete: {summary}`
   - Add row: Elective agent failed → broadcast warning → `[ORCHESTRATOR] {agent_name} failed: {summary}`
   - Add row: Elective agent blocked → broadcast warning → `[ORCHESTRATOR] {agent_name} deferred — Ship shipment {id} is active`

**Acceptance criteria**:
- Step 1E is clearly positioned as an alternate routing path, not part of the sequential Stage→Ship flow
- Concurrency constraints are unambiguous: elective blocked during Ship, allowed during Stage
- Subagent depth correctly reflects the shallower tree (max 2 vs 3)
- Intercom rows follow the existing pattern (When / Tool / Level / Message)
- No unresolved template variables; all {{VARIABLES}} follow convention
