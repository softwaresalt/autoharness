# Spike: PR Comment Escape-Character RCA

> **Status: complete.** Root cause identified and a concrete instruction-template
> fix delivered alongside this findings doc.

## Symptom

Code-review comment *resolution* replies (the "Fixed in `<sha>` …" and
"Declined — …" comments the Ship agent posts back to Copilot review threads)
intermittently contain corrupted text: dropped characters, swapped/empty words,
and truncated inline code spans. Operators described these as "a lot of typos
that seem related to escape characters."

## Root Cause

The corruption is **not** produced by the model writing bad text. It is
introduced by the **shell transport** when a reply body is handed to `gh api`
through an inlined, double-quoted **PowerShell** string:

```powershell
gh api repos/OWNER/REPO/pulls/<pr>/comments/<id>/replies -X POST -f body="Fixed in df186ac. Removed the unused `types` import."
```

PowerShell pre-processes a double-quoted string *before* `gh` ever receives it:

1. **Backtick is the PowerShell escape character.** Review replies almost always
   contain markdown code spans (`` `types` ``, `` `--strict` ``,
   `` `size_task` ``). Each backtick is interpreted as an escape operator:
   PowerShell consumes it and escapes the following character. `` `types` ``
   collapses toward `types` (backticks gone) and, worse, `` `n ``/`` `t ``
   sequences inside a code span become a newline/tab. This is the dominant
   source of the "typos."
2. **`$` triggers variable expansion.** A body mentioning `$result`, `$PATH`, or
   `$1` expands to an empty or unexpected value inside a double-quoted string.
3. **Embedded double quotes terminate the argument early**, truncating the
   remainder of the comment.

Because review replies are the one place the workflow routinely emits markdown
containing backticks and `$`, they collide with PowerShell's two most aggressive
metacharacters far more often than ordinary commit messages — which is why the
symptom clusters on *comment resolution* specifically.

### Why it is intermittent

Replies without backticks, `$`, or embedded quotes pass through unharmed (e.g.,
the plain "Fixed in df186ac. Removed the unused import." variant). The defect
only manifests when the body carries one of those metacharacters, so the same
workflow looks healthy on some PRs and corrupted on others.

## Contributing Factor

The instruction templates already recommend the structured MCP tool
`mcp_github_add_reply_to_pull_request_comment` (shell-safe), but they did not
warn against the `gh api ... -f body="..."` fallback, nor document a safe
fallback transport. When the MCP tool was unavailable, agents reached for the
inline `gh api` form and reintroduced the hazard.

## Fix

Add a **Shell-Safe Comment Body Construction** guardrail to
`github-pr-automation.instructions` (both the `templates/` source and the
installed `.github/` mirror), directly under §1.5 "Reply to Addressed Comments":

* Prefer the structured MCP tool (no shell parsing).
* When `gh api` must be used, never inline the body in a double-quoted
  PowerShell string (and never even *construct* the body as a double-quoted
  string — corruption happens at construction). Write the exact body to a
  **BOM-less** UTF-8 file and pass it by reference with `--field body=@file`
  (`-F`, not `-f`, whose `@file` is sent literally); this reads the file as a
  raw string with no interpolation and no type coercion, and works for REST
  replies and any GraphQL mutation carrying a markdown body. A single-quoted
  here-string is a safe inline alternative (literal — no backtick escaping, no
  `$` expansion) except when the body contains a line that is exactly `'@` at
  column 0.
* Never hand-escape or strip markdown backticks to "make the command work" —
  change the transport, not the content.

> **Portability note:** PowerShell is the observed root cause in this workflow,
> but the same inline-interpolation hazard exists in Bash/Zsh (backtick command
> substitution, `$` expansion). On Windows PowerShell 5.1, `Set-Content
> -Encoding utf8` emits a UTF-8 BOM that would prepend a stray U+FEFF to the
> posted comment — hence the BOM-less `.NET` writer in the guidance.

## Verification

* markdownlint passes on both edited instruction files.
* `verify_workspace` template/mirror parity check remains green (guidance added
  symmetrically to template and mirror).
* No code paths change; this is an instruction/guardrail hardening.

## Follow-ups

* The same hazard applies to `git commit -m "..."` bodies on PowerShell when a
  commit message contains backticks/`$`. Out of scope for this spike (operator
  symptom was specifically comment resolution), but a candidate future guardrail
  in the commit/PR-body instructions.
