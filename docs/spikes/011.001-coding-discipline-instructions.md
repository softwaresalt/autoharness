# Spike Findings: karpathy-guidelines as Coding Discipline Instructions

## Summary of karpathy-guidelines

ATV's `karpathy-guidelines` skill encodes four behavioral rules derived from Andrej Karpathy's observations on common LLM coding pitfalls:

| Rule | Core Directive |
|---|---|
| **1. Think Before Coding** | Surface assumptions explicitly; push back on ambiguity; don't pick silently |
| **2. Simplicity First** | Minimum code that solves the problem; no speculative features, abstractions, or "flexibility" |
| **3. Surgical Changes** | Touch only what the request requires; don't improve adjacent code; remove only orphans you created |
| **4. Goal-Driven Execution** | Transform tasks into verifiable goals; state a brief plan with per-step success criteria |

The source is a SKILL.md, but the content is entirely declarative behavioral rules — there are no workflow steps, tool calls, or skill invocations. It functions as an instruction file.

## Gap Analysis vs. autoharness Constitution

| karpathy Rule | autoharness Constitution Coverage | Gap |
|---|---|---|
| Think Before Coding | No direct equivalent. Principle II (Test-First) touches "define success criteria" but only in the context of TDD, not as a general pre-coding discipline. | **Gap**: No explicit "surface assumptions, resist ambiguity, ask before implementing" instruction at the coding level. |
| Simplicity First | Principle VI (Single Responsibility) covers "no speculative dependencies" but only for external libraries, not for code itself. | **Gap**: No explicit "no abstractions for single-use code, no flexibility that wasn't requested" rule. The constitution guards the dependency graph; karpathy guards the code itself. |
| Surgical Changes | Not covered. The constitution says nothing about how to scope edits when modifying existing code. | **Gap**: This is entirely missing from autoharness instructions. It directly addresses the common LLM anti-pattern of "improving" adjacent code while fixing something else. |
| Goal-Driven Execution | Principle II (Test-First) enforces test-before-code and red-green verification. | **Partial coverage**: Principle II covers the TDD dimension. The karpathy formulation is broader (applies to all tasks, not just code tasks) and adds the per-step plan with explicit success criteria. |

**Verdict**: The karpathy rules operate at a different layer from the constitution. The constitution governs *what* to build (architecture, safety, process). The karpathy rules govern *how* to write code (cognitive discipline, scope management, verification). They are **complementary, not overlapping** — adoption would deepen Primitive 6 without conflicting with existing content.

## Recommendation: Adopt

Create `templates/instructions/coding-discipline.instructions.md.tmpl` with `applyTo: '**'`.

**Rationale**:
- High signal-to-noise: every rule targets a real LLM failure mode
- No duplication: fills a gap the constitution explicitly does not cover
- Zero parameterization needed: the rules are behavioral and technology-agnostic
- Template weight: ~60 lines — acceptable for `applyTo: '**'` scope

**Proposed template skeleton**:

```markdown
---
description: "Coding discipline — behavioral rules to reduce common LLM coding mistakes"
applyTo: '**'
---

# Coding Discipline

Behavioral rules to reduce common LLM coding mistakes. Apply to all coding tasks.

**Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
\`\`\`
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
\`\`\`

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.
```

**applyTo scope**: `'**'` — these rules apply universally and are load-on-demand via the injection point mechanism.

**No template variables needed**: all rules are behavioral and technology-agnostic. The only optional parameterization would be `{{DOCS_MEMORY}}` in the log path if a logging hook were added — not needed for the base template.
