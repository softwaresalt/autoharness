# Spike Charter: Reference Adoption Evaluation

> **Status: complete.** This spike sampled both reference repositories and records
> adoption guidance only. No template or code changes are made by this spike.

## Objective

Evaluate two reference repositories present in the workspace for patterns,
skills, workflow surfaces, and capabilities that autoharness could **adopt**,
**adapt**, or **reject**, assessed against the 10-primitives + capability-pack
model.

Both reference paths are confirmed present:

* `references/mattpocock-eng-skills`
* `references/atv-starterkit`

## Scope and Method (time-box ~2h)

1. Inventory the notable skills/agents/workflows/patterns in each reference repo
   (sample representative surfaces rather than exhaustively cataloguing large
   trees).
2. For each notable pattern, classify as **adopt** / **adapt** / **reject** with:
   * a one-line description of the pattern,
   * which primitive(s) or capability pack it maps to (or whether it implies a
     new overlay),
   * rationale for the classification,
   * concrete follow-on work if adopt/adapt.
3. Note any overlap with already-shipped autoharness capabilities (avoid
   re-adopting patterns already covered — e.g., the `atv-starterkit` portability
   audit pattern was already evaluated in `011.004-T`).

## Constraints

* **Investigation only.** Read-only. Do not modify `templates/`, `schemas/`,
  `.github/`, or `src/`.
* Follow-on work is captured as **new stash entries**, not implemented here.
* Respect the capability-pack overlay contract: any proposed pack must define
  eligibility signals, recommendation logic, overlay targets, behavior deltas,
  verification checks, and tuning drift checks.

## Findings — mattpocock-eng-skills

### Invocation taxonomy and bucket discipline — adapt later

`references/mattpocock-eng-skills/docs/invocation.md` defines a crisp
user-invoked vs model-invoked split:

* user-invoked skills set `disable-model-invocation: true` and are reachable
  only by explicit human command;
* model-invoked skills keep rich trigger descriptions so the model can reach for
  them automatically;
* a user-invoked skill may invoke model-invoked skills, but not another
  user-invoked skill.

This maps to Primitive 4 (orchestration) and Primitive 6 (injection points). It
is worth adapting if autoharness later introduces first-class user-only prompts
versus auto-invokable skills, but it is not an immediate adoption because
autoharness already separates agents, prompts, and skills through frontmatter and
install layers.

### Grilling and domain-modeling loop — adapt into future brainstorm/domain overlay

`grill-with-docs` is intentionally tiny: it runs the reusable `grilling` loop
with the `domain-modeling` skill. The underlying domain-modeling skill actively
sharpens glossary terms, stress-tests scenarios, and writes context/ADR material
only when the decision is hard to reverse, surprising without context, and the
result of a real trade-off.

This is a strong fit for Primitive 2 (scope clarity), Primitive 9 (repository
knowledge), and a possible future `domain-glossary` or `brainstorm-plus` overlay.
It overlaps with autoharness' newly shipped brainstorm-led intake and deliberate
workflow, so it should be adapted later only when a concrete workspace needs
domain-language capture.

### Vertical-slice issue generation — already covered by harvest/impl-plan

`to-issues` emphasizes tracer-bullet vertical slices, dependency ordering, and
issue bodies with acceptance criteria. autoharness already has the same core
shape through `impl-plan`, `harvest`, task granularity, and shipment assembly.
Keep the wording as a reference, but do not adopt another issue-generation skill.

### Deep-module design vocabulary — adapt as reviewer vocabulary, not a new core primitive

`codebase-design` provides useful vocabulary: module, interface, seam, adapter,
depth, leverage, and locality. This is relevant to Primitive 7 review personas
and Primitive 9 repository legibility. It should be considered as future
reviewer vocabulary for architecture/design reviews, not as a new primitive or
workflow stage.

## Findings — atv-starterkit

### Brainstorm -> plan -> work -> review -> compound pipeline — already partly adopted

ATV's README and `ce-*` skills model a full lifecycle:

```text
/ce-brainstorm -> /ce-plan -> /ce-work -> /ce-review -> /ce-compound
```

autoharness already maps this into Stage -> Ship with deliberate/brainstorm
intake, implementation planning, review, PR lifecycle, and compound/closure
knowledge capture. The recently shipped dark-factory work explicitly borrowed
the `/ce-brainstorm` design for bounded requirements before autonomous handoff.
No new action is needed from this spike.

### `lfg` resumable run-state helper — reject for autoharness core

ATV's `/lfg` uses local `.atv/runs/<run-id>/` sentinels to resume an autonomous
pipeline. autoharness already uses backlogit state, shipment records, PR
readiness blocks, closure artifacts, and session memory. Adding a second local
run-state system would duplicate state and make P-001/P-016 reasoning harder.
Reject for core autoharness.

### Dual-review + fix/reply/resolve workflow — already incorporated into dark mode

`ghcp-review-resolve` validates a dual-review/fix loop where fixes are committed
and pushed before replies, and bot-authored threads are resolved through GraphQL.
This pattern was already incorporated into the dark-mode operating-rule work and
GitHub PR automation guidance. No new follow-up is needed.

### Agent and skill catalog breadth — reject wholesale adoption

ATV ships a large catalog of agents and skills. autoharness should not copy that
catalog wholesale. The capability-pack overlay model requires eligibility
signals, recommendation logic, overlay targets, behavior deltas, verification
checks, and tuning drift checks before any pack is promoted. Individual ideas
remain useful as research input, but direct catalog import would violate the
global-tool/local-output architecture.
