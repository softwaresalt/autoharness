# Spike Findings: SDK Intelligence Layer Patterns for Guardrail Templates

## ATV Pattern Summary

`atv-starterkit/pkg/sdk/intelligence.go` implements a three-layer resilience system for SDK queries:

### Layer 1: Rate Limiting — Minimum Interval

```go
defaultMinInterval = 30 * time.Second
```

Every query is gated by a minimum interval: if less than 30 seconds has elapsed since the last query, the call returns `nil` silently. This prevents burst traffic from exhausting quotas.

**Test coverage**: `TestRateLimit_MinInterval` — sets interval to 1 hour, confirms second query is rate-limited.

### Layer 2: Rate Limiting — Hourly Budget

```go
defaultQueryBudget = 60 // Max queries per hour
```

A rolling hourly window tracks `queriesUsed`. When the budget is exhausted, queries return `nil` silently until the next hour window. The window rolls at clock-hour boundaries (`time.Now().Truncate(time.Hour)`).

**Test coverage**: `TestRateLimit_Budget` — sets budget to 2, confirms third query is budget-limited.

### Layer 3: Circuit Breaker

```go
circuitBreakerLimit = 3  // consecutive failures to trip
circuitBreakerCool  = 5 * time.Minute // cooldown before auto-reset
```

After 3 consecutive `recordFailure()` calls, `circuitOpen = true` and `circuitOpenUntil = now + 5min`. Queries return `nil` while the circuit is open. After the cooldown expires, the next `Query()` call auto-resets the circuit (`circuitOpen = false`, `consecutiveFailures = 0`) and attempts the query.

`recordSuccess()` resets `consecutiveFailures` to 0 but does **not** close an already-open circuit — only the cooldown timer does that.

**Test coverage**: `TestCircuitBreaker`, `TestCircuitBreaker_Reset`.

### Offline Degradation

The `Start()` method sets `online = false` if SDK init fails. All operations (`Query`, `Explain`) check `online` first and return nil/error without counting as failures. This keeps the circuit breaker stats clean during known-offline periods.

### concurrency.instructions.md.tmpl

ATV's intelligence layer has no concurrency patterns (uses `sync.RWMutex` internally, but that's implementation detail). The concurrency template covers file locking for multi-agent scenarios. **No gap here** — the SDK patterns don't inform concurrent file access.

### Key Finding: Time-Based Cooldown is Adoptable

The most transferable pattern from ATV is the **time-based cooldown + auto-reset** for the circuit breaker. autoharness currently says:

> "3. Prompt — surface the following message to the operator: `Circuit breaker triggered after {N} consecutive failures.`"
> "4. Checkpoint — write a memory checkpoint so session state is preserved."

This implies the circuit stays open indefinitely until the operator responds. In practice, many transient failures (network hiccups, temporary tool unavailability) resolve within minutes. The ATV pattern of a 5-minute cooldown with automatic retry recovery is:

1. Directly applicable to agent session flows
2. Consistent with the spirit of the template (avoid infinite loops, not infinite operator waits)
3. Concretely specified and testable

The **rate limiting patterns** (MinInterval, QueryBudget) are **not directly applicable** to the general circuit breaker template. They're specific to SDK query throttling — a use case where an external API charges per query or enforces rate limits. autoharness circuit breaker addresses retry loops and tool failures, not query budgets.

## Recommendation: Partial Adopt

**Adopt**: Add time-based cooldown + auto-reset to `circuit-breaker.instructions.md.tmpl` as an optional pattern.

**Defer**: Rate limiting (min interval, hourly budget) patterns — these are SDK/API throttling patterns, not general circuit breaker patterns. They don't belong in a universal circuit breaker instruction template.

### Proposed Template Delta

Add a new "Cooldown and Auto-Reset" section to `circuit-breaker.instructions.md.tmpl`:

```markdown
## Cooldown and Auto-Reset (Optional)

For transient failures where the underlying cause is likely to resolve automatically
(network hiccups, temporary tool unavailability, rate limit windows), agents MAY
implement a time-bounded cooldown rather than halting immediately for operator input:

1. On circuit trip: record `circuit_open_until = now + {{CIRCUIT_BREAKER_COOLDOWN}}` (default: 5 minutes)
2. On subsequent calls while circuit is open and `now < circuit_open_until`: return degraded/nil immediately
3. When `now >= circuit_open_until`: auto-reset (`circuit_open = false`, `consecutive_failures = 0`), retry once
4. If retry fails: re-trip the circuit (same 3-failure threshold applies), double the cooldown period
5. After 3 circuit trips: escalate to operator — do not continue auto-resetting

Cooldown is appropriate when:
- The operation is non-critical (context enrichment, optional tool calls)
- Failures are consistent with transient conditions (timeouts, rate limits)
- The operator is not expected to be present (automated/unattended sessions)

Cooldown is NOT appropriate when:
- The failure indicates a logic error (wrong arguments, auth failure, schema mismatch)
- The operation is a critical gate (index sync, shipment claim, PR merge)
- 3+ circuit trips have already occurred in the session
```

**New template variable**: `{{CIRCUIT_BREAKER_COOLDOWN}}` (default: `5 minutes`)

This variable would appear in the install-harness variable resolution table and would default to `5 minutes` for all workspaces unless the workspace profile specifies otherwise.
